# skill_07_html_components.md

## Библиотека контентных блоков

### Назначение
Агент не придумывает HTML-разметку.
Агент выбирает нужный тип блока из этой библиотеки
и заполняет его контентом.

Выбор типа блока — на усмотрение агента исходя из контекста.
Шаблон блока — фиксированный, менять нельзя.

---

## Правила выбора блока

### Когда использовать каждый тип:

**paragraph** — основной текст раздела.
Всегда. Базовый блок для любого абзаца.

**fact** — используй когда:
- Есть конкретная цифра или статистика
- Есть неочевидный факт который удивит читателя
- Хочется выделить ключевую мысль раздела
- Пример: "Число 7 — единственное простое число среди базовых чисел нумерологии"

**quote** — используй когда:
- Есть сильное утверждение которое стоит выделить визуально
- Формулировка настолько точная что её стоит процитировать
- Нужен эмоциональный якорь в середине длинного раздела
- Пример: "Семёрка не ищет одобрения — она ищет истину"

**table** — используй когда:
- Нужно сравнить 3 и более элементов
- Есть структурированные данные с несколькими параметрами
- Совместимость чисел, сравнение систем нумерологии
- НЕ используй для простых списков из 2 колонок — там достаточно ul

**list** — используй когда:
- Перечисление из 3+ однородных элементов
- ul: порядок не важен (качества, характеристики, примеры)
- ol: порядок важен (шаги, этапы, последовательность)

**callout** — используй когда:
- Нужно честно обозначить ограничение системы
- Есть важное замечание которое нельзя пропустить
- Предупреждение о распространённом заблуждении
- Три варианта: info (нейтральное), tip (совет), warning (предупреждение)

**fact_row** — используй когда:
- Нужно показать несколько фактов в ряд
- Краткие характеристики числа (3-4 параметра)
- Пример: энергия / стихия / планета / цвет

## Типы блоков и их JSON-схема

### paragraph
```json
{
  "type": "paragraph",
  "text": "текст абзаца"
}
```

### fact
```json
{
  "type": "fact",
  "text": "текст факта — конкретно и по существу"
}
```

### quote
```json
{
  "type": "quote",
  "text": "текст цитаты или сильного утверждения",
  "source": "источник или контекст (необязательно)"
}
```

### table
```json
{
  "type": "table",
  "caption": "заголовок таблицы",
  "headers": ["Колонка 1", "Колонка 2", "Колонка 3"],
  "rows": [
    ["значение 1", "значение 2", "значение 3"],
    ["значение 1", "значение 2", "значение 3"]
  ]
}
```

### list
```json
{
  "type": "list",
  "style": "ul",
  "items": [
    "пункт 1",
    "пункт 2",
    "пункт 3"
  ]
}
```
Для нумерованного: "style": "ol"

### callout
```json
{
  "type": "callout",
  "variant": "info",
  "text": "текст блока"
}
```
Варианты: "info" / "tip" / "warning"

### fact_row
```json
{
  "type": "fact_row",
  "items": [
    {"label": "Энергия", "value": "аналитика и поиск"},
    {"label": "Стихия", "value": "вода"},
    {"label": "Планета", "value": "Нептун"},
    {"label": "Цвет", "value": "фиолетовый"}
  ]
}
```

**Рендер fact_row в читаемой части статьи (ЧАСТЬ 1):**
Выводится строкой через разделитель — не таблицей:
```
**Энергия:** аналитика и поиск | **Стихия:** вода | **Планета:** Нептун | **Цвет:** фиолетовый
```
Не используй markdown-таблицу для fact_row.

### h2
```json
{
  "type": "h2",
  "text": "заголовок раздела",
  "lead": "эмоциональный абзац после H2 — обязателен"
}
```

**Lead после H2:**
- Объём: 3-4 предложения, 70-90 слов
- Создаёт образ, задействует воображение
- Числа как живые сущности
- Чередование коротких и длинных предложений

### h3
```json
{
  "type": "h3",
  "text": "заголовок подраздела"
}
```

### cta
```json
{
  "type": "cta",
  "position": "after_intro | after_calculation | end_of_article | mid_article_channel",
  "heading": "заголовок баннера — 5-8 слов, конкретно и по теме статьи",
  "subtext": "подзаголовок — одно предложение, усиливает заголовок",
  "button_text": "текст кнопки →",
  "url": "https://t.me/numerolog_master_bot или https://t.me/chisla_vlasti"
}
```

Позиции:
- `after_intro` — после вводного абзаца, ссылка на бота
- `after_calculation` — после раздела с расчётом, ссылка на бота
- `end_of_article` — в конце статьи, ссылка на бота
- `mid_article_channel` — в середине статьи, ссылка на канал

Правила:
- Максимум 2 CTA на бота с разными `heading` и `button_text`
- Ровно 1 CTA на канал (`mid_article_channel`)
- `heading` и `subtext` релевантны теме конкретной статьи
- Ссылка на бота: `https://t.me/numerolog_master_bot`
- Ссылка на канал: `https://t.me/chisla_vlasti`

### faq
```json
{
  "type": "faq",
  "items": [
    {
      "question": "Вопрос в живом языке?",
      "answer": "Ответ 2-4 предложения."
    }
  ]
}
```

### links
```json
{
  "type": "links",
  "title": "Изучите подробнее",
  "items": [
    {
      "url": "/chislo-sudby/1/",
      "anchor": "Число судьбы 1",
      "description": "путь лидера и первопроходца"
    }
  ]
}
```

## Полная структура поля blocks в JSON статьи

Поле `blocks` заменяет отдельные поля `structure` и `faq`.
Агент формирует `blocks` как массив блоков в порядке появления в статье.

### Обязательный порядок блоков:

```
1. intro (paragraph)
2. cta after_intro
3. h2 + lead
4. h3
5. paragraph / fact / quote / table / list / callout
   (агент выбирает сам исходя из контекста)
6. ... повторяется для каждого раздела ...
7. faq
8. links (для hub)
9. cta end_of_article
```

### Правила компоновки:

**Минимум на статью:**
- 1 fact
- 1 quote
- 1 table
- 1 list
- fact_row — ровно 1 раз, только в первом разделе H2

**Структура раздела H3:**
Раздел H3 может разбиваться на два абзаца с элементом внимания между ними — fact или quote.
Применяй не чаще чем в каждом втором H3.
Выбирай те разделы где есть реально сильный факт или точная формулировка —
не вставляй элемент ради структуры.

**Ограничения:**
- callout — не чаще 1 раза на каждые 2 раздела H2
- fact и quote не используются оба в одном разделе H3;
  один из них допустим между двумя абзацами внутри H3
- table и list — никогда не стоят в одном разделе H3

**Обязательные разделители:**
- Между любыми двумя специальными блоками (fact, quote, table, list, callout, fact_row)
  всегда должен стоять минимум один paragraph
- Специальные блоки никогда не идут подряд без параграфа между ними

**Правильно:**
```
h3 → paragraph → fact → paragraph → [callout / table / list — опционально]
h3 → paragraph → paragraph (без элемента — тоже нормально)
```

**Неправильно:**
```
fact → quote
table → list
fact → fact
paragraph → fact → quote (два элемента в одном H3)
```

### Пример скомпонованного раздела:

```json
{"type": "h3", "text": "Характер и таланты"},
{"type": "paragraph", "text": "первый абзац — основной текст"},
{"type": "fact", "text": "Люди с числом судьбы 7 чаще других выбирают..."},
{"type": "paragraph", "text": "второй абзац — продолжение мысли"},

{"type": "h3", "text": "Карьера и деньги"},
{"type": "paragraph", "text": "единственный абзац — элемент внимания не нужен"},
{"type": "table", "caption": "Профессии для числа 7", "headers": ["Сфера", "Профессии", "Почему подходит"], "rows": [["Наука", "Исследователь, аналитик", "Глубокое мышление"]]}
```

## Чего НЕ делает агент

- Не пишет HTML-теги напрямую
- Не придумывает новые типы блоков
- Не меняет схему существующих блоков
- Не ставит source в quote если источника нет — просто опускает поле
- Не использует markdown-таблицу для рендера fact_row в читаемой части статьи
- Не вставляет fact или quote между абзацами H3 ради структуры — только если есть реально сильный материал
