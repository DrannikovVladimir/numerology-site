# Настройка SSH-доступа к серверу на новой машине

Сервер: `185.113.132.85` (chislavlasti.com)
Пользователь: `deploy` (root-вход по SSH отключён)

После хардеинга SSH (см. `docs/DEPLOYMENT.md`) вход на сервер возможен
**только по ключу**, под пользователем `deploy`. Пароль и root-логин
отключены. Ниже — что нужно перенести/настроить на новой машине, чтобы
получить доступ.

## Вариант A: перенести существующий ключ (рекомендуется)

Самый простой путь — скопировать уже рабочий ключ с текущей машины на новую.
Ключ действует одинаково с любой машины, если он физически перенесён.

**На старой машине** (там, где ключ уже настроен), забери два файла:
```
~/.ssh/id_ed25519_deploy       ← приватный ключ, НИКОГДА никому не показывать
~/.ssh/id_ed25519_deploy.pub   ← публичный ключ, можно копировать открыто
```

Перенести их можно любым надёжным способом (не через публичный чат/облако
без шифрования): флешка, `scp` между машинами напрямую, менеджер паролей
с поддержкой файлов, зашифрованный архив.

**На новой машине** — положить оба файла в `~/.ssh/`:
```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
# скопировать оба файла в ~/.ssh/
chmod 600 ~/.ssh/id_ed25519_deploy
chmod 644 ~/.ssh/id_ed25519_deploy.pub
```

Права важны — SSH откажется использовать приватный ключ с слишком
открытыми правами доступа.

Дальше — сразу к разделу **«Настройка удобного алиаса»** ниже, ключ уже
знаком серверу (он там же в `authorized_keys`), генерировать новый и
добавлять на сервер не нужно.

## Вариант B: сгенерировать новый ключ для новой машины

Более безопасный вариант, если старая машина ненадёжна или ключ переносить
не хочется. У каждой машины будет свой отдельный ключ — если одну потеряешь
или скомпрометируешь, достаточно отозвать только её ключ на сервере,
остальные продолжат работать.

**На новой машине:**
```bash
ssh-keygen -t ed25519 -C "имя-новой-машины" -f ~/.ssh/id_ed25519_deploy
```
Enter на вопрос про passphrase (или задать свой — на усмотрение).

Вывести публичный ключ и скопировать его целиком:
```bash
cat ~/.ssh/id_ed25519_deploy.pub
```

**На сервере** — понадобится доступ с уже настроенной машины (или через
консоль провайдера, если доступа больше нет). Под пользователем `deploy`:
```bash
echo "СЮДА_ВСТАВИТЬ_СКОПИРОВАННЫЙ_ПУБЛИЧНЫЙ_КЛЮЧ" >> ~/.ssh/authorized_keys
```

Проверить, что добавилось верно:
```bash
cat ~/.ssh/authorized_keys
```
Должна появиться новая строка рядом с уже существующими ключами
(`github-actions-deploy`, `krowen-personal` и т.д.) — старые строки трогать
не нужно, просто дописывается новая.

## Настройка удобного алиаса (оба варианта)

Чтобы не писать каждый раз полную команду с `-i`, добавь на новой машине
в `~/.ssh/config`:
```
Host numerology-server
    HostName 185.113.132.85
    User deploy
    IdentityFile ~/.ssh/id_ed25519_deploy
```

После этого подключение — просто:
```bash
ssh numerology-server
```

## Проверка доступа

```bash
ssh numerology-server
sudo whoami
```
Первая команда должна пустить без пароля (вопрос про passphrase ключа —
это нормально, если ты его задавал при генерации). Вторая — запросит
пароль пользователя `deploy` и выведет `root`.

## Важно помнить

- **Root по SSH отключён полностью** — `ssh root@185.113.132.85` не
  сработает ни с одной машины, это ожидаемое поведение
- **Вход по паролю отключён** — если ключ утерян или недоступен, вход
  на сервер возможен только через web-консоль провайдера сервера
  (панель управления VPS)
- Приватный ключ (`id_ed25519_deploy`, без `.pub`) — секрет, эквивалент
  пароля. Не пересылать открытым текстом, не класть в облачные заметки
  без шифрования, не коммитить в git
- Если машина/ключ скомпрометированы или больше не нужны — удали
  соответствующую строку из `~/.ssh/authorized_keys` на сервере (под
  `deploy`), это мгновенно отзывает доступ для этого ключа, остальные
  не затрагиваются
