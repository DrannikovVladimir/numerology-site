# Автопилот

## Два режима работы

**Генерация батча** — не реализована как скрипт. Сейчас статьи генерируются
вручную, через чат, по промптам и скилам из `/autopilot/prompts/`. Готовый
файл сохраняется вручную в `/content/pending/`.

**Публикация** — реализована (`autopilot/publish.js`), запускается вручную
или по расписанию через cron. Берёт файлы из `/content/pending/` и публикует
на сайт.

## Генерация батча

Скрипт `generate.js` не реализован. Генерация статьи сегодня — это ручной
прогон промптов `prompt_02_onpage.md` → `prompt_03_generator.md` в чате, с
подключением нужных skill-файлов (базовый `skill_05_base.md` + файл серии
из `/spoke/` или структура хаба/standalone), и ручное сохранение результата
в `/content/pending/{page_id-с-дефисами}.json`.

Ниже — целевой интерфейс скрипта на случай, если/когда он будет написан:
```bash
node autopilot/generate.js --type spoke --limit 50
node autopilot/generate.js --type standalone --limit 13
```
```
--type    тип страниц: spoke / standalone / all
--limit   максимум статей за запуск
--from    начать с конкретного кластера (опционально)
```

## Публикатор

### Запуск
```bash
# crontab — по одной статье в сутки
0 10 * * * node /autopilot/publish.js
```
```bash
node autopilot/publish.js              # 1 статья — обычный режим для cron
node autopilot/publish.js --count 10   # N статей разом — разовый запуск
                                        # (например, стартовый пул статей в день деплоя)
node autopilot/publish.js --dry-run    # показать план публикации, не трогая файлы на диске
```

### Алгоритм
```
1. Прочитать все .json из /content/pending/, отсортировать по mtime файла
   (FIFO — старейшие первыми; НЕ по created_at внутри самого JSON)
2. Взять первые N файлов (N = --count, по умолчанию 1)
На каждый файл:
  а. Если date_published ещё не заполнен → проставить сегодняшнюю дату (YYYY-MM-DD)
  б. date_modified — проставить сегодняшнюю дату всегда
  в. Переложить файл из /pending/ в /published/
3. После обработки всех файлов — вызвать update-planned-urls.js
   (полностью пересобирает planned-urls.json на основе того, что реально
   лежит в /content/published/, а не дописывает в него)

Если /content/pending/ пуст — выход без ошибки.
```

Скрипт **не делает**:
- не проверяет наличие изображения в `/content/images/` — эта логика на
  стороне рендера сайта (`article.image && (...)` в `app/[...slug]/page.tsx`);
  статья без картинки публикуется как есть, JSON-LD корректно опускает поле `image`
- не обновляет статус кластера в `semantic_clusters.json`
- не отправляет уведомления в Telegram — `telegram.js` не реализован, флага
  `--notify` в скрипте нет
- не запускает `linkbuilder.js` — линковка отдельный шаг, выполняется вручную
  после публикации:
  ```bash
  node autopilot/linkbuilder.js --file content/published/{имя-файла}.json
  ```

### Обновление sitemap.xml
Отдельным шагом publish.js не делает и делать не должен. `sitemap.xml` —
статический роут (`○ Static` в выводе `next build`), собирается один раз при
`npm run build` на основе `planned-urls.json` на тот момент. Новый URL,
опубликованный после сборки, не появится в `sitemap.xml`, пока сайт не
пересобран заново. Проверено вручную: в `npm run dev` новый URL виден в
sitemap сразу же (Next.js Fast Refresh переоценивает модуль на лету); в
`npm run build && npm start` — не появляется без пересборки. Для VPS это
означает: после публикации новых статей нужна периодическая пересборка
(`npm run build && pm2 restart site`), если важно, чтобы sitemap был
актуален для поисковиков.

### Уведомление в Telegram
Не реализовано. `telegram.js` не написан, `publish.js` не поддерживает
`--notify`. Формат сообщения ниже — целевой, на случай будущей реализации:
```
✓ Опубликована новая статья

Название: Число судьбы 7 — путь мудреца
URL: /chislo-sudby/7/
Тип: spoke
Осталось в очереди: 47
```

## Обработка ошибок

| Ситуация | Действие |
|----------|----------|
| /pending/ пуст при публикации | Выход без ошибки |
| Изображение отсутствует | Публикуется без изображения — статья рендерится
                             без `<figure>`, JSON-LD не включает `image` |
| Ошибка чтения/записи JSON в publish.js | Файл пропускается, остаётся в
                                            pending, обработка остальных
                                            файлов продолжается |

## linkbuilder.js — автоматическая простановка контекстных ссылок

**Место в пайплайне:** после ручной генерации статьи, перед или после
перекладывания в `/pending/`/`/published/` — порядок не жёсткий, скрипт
работает с файлом на месте (`--file`) или с папкой целиком (`--dir`).
Также используется вручную для ретроактивной простановки на уже
опубликованные статьи.

### Запуск
```bash
node autopilot/linkbuilder.js --file content/pending/chislo-sudby-7.json
node autopilot/linkbuilder.js --file content/published/chislo-sudby-7.json
node autopilot/linkbuilder.js --dir content/published/
node autopilot/linkbuilder.js --dir content/published/ --dry-run
```
- `--file` — обработать один файл
- `--dir` — обработать все `.json` в папке (ретроактивная простановка,
  проверено на всех 33+1 опубликованных статьях без единой ошибки выполнения)
- `--dry-run` — не перезаписывать оригинал; сохранить рядом `{filename}.linkbuilder-preview.json`

### Словарь анкоров — `content/anchors.json`
Не редактируется руками. Перед каждым запуском `linkbuilder.js` автоматически
вызывает `build-anchors.js`, который пересобирает этот файл заново из
`content/semantic_clusters.json` (только кластеры со статусом `true` в
`planned-urls.json`) + `russian-nouns-js` (падежные формы существительных)
+ встроенная таблица числительных-синонимов (единица, двойка, тройка... —
только для шаблона `chislo-sudby-1-9`, только существительные, без голых
числительных — риск ложных совпадений на обычных словах языка).
Можно запустить отдельно: `node autopilot/build-anchors.js`.

### Алгоритм
1. Собрать словарь анкоров (см. выше)
2. Собрать URL, уже занятые в статье — из инлайн-ссылок `[текст](url)` в
   параграфах, `internal_links.up.url`, и всех `url` в блоке `links`
   ("По теме") — эти URL нигде дальше не предлагаются повторно
3. Разбить статью на зоны: `intro` (до первого `h2`), и по одной зоне на
   каждый `h2` до следующего `h2` или конца статьи (`h3` не образует
   отдельную зону, входит в зону родительского `h2`)
4. Зона, где уже есть хотя бы одна ссылка — пропускается целиком
5. В открытой зоне — анкоры (за вычетом уже занятых URL) сортируются от
   длинных к коротким, ищется точное совпадение подстроки без учёта
   регистра, с проверкой границ слова для кириллицы; макс. 2 ссылки на зону
6. Результат — консольный отчёт по каждой зоне + сохранённый файл

Матчинг работает по чистой подстроке текста, без понимания контекста —
на практике это оказалось надёжнее, чем ожидалось: в статье про 22:44
слово «тройка» матчилось на `/chislo-sudby/3/` корректно, потому что текст
сам объясняет числовую редукцию (2+2+4+4=12 → 1+2=3), а не как омоним
«группа из трёх». Ложных срабатываний за 34 обработанных статьи не найдено.

### Известные ограничения (на заметку для следующей итерации)
- Зонирование по `h2` даёт неравномерную плотность — в статье с несколькими
  `h3` внутри одного `h2` ссылки могут скучиваться в верхней части раздела,
  до первого подзаголовка. Рассматривается переход на зонирование по `h3`
  с лимитом 1 ссылка на зону — требует также общего потолка ссылок на
  статью целиком, чтобы длинные статьи с большим числом `h3` не получали
  ссылок больше, чем нужно. Решение не принято.
- Анкоры match'ятся только в точной форме, полученной от `russian-nouns-js`
  (падежи) и таблицы синонимов — без синонимов вне этого набора.
- Текст хлебной крошки на хаб (`internal_links.up.anchor`) генерируется
  заново промптом 2/3 в каждой статье, а не берётся из `nav_title`
  hub-файла — вызывает разный регистр текста крошки для одного и того же
  хаба между статьями (например «числа судьбы» vs «Ангельские числа»). Это
  не зона ответственности `linkbuilder.js` — чинить нужно в промпте/скиле
  генерации.
