# Архитектура системы

## Общая схема

```
┌─────────────────────────────────────┐
│         РУЧНАЯ ГЕНЕРАЦИЯ             │
│   Промпт 2 → Промпт 3 (чат с Claude) │
│      → локально, content/pending/    │
└──────────────┬──────────────────────┘
               ↓ rsync (scripts/sync-content.sh)
┌─────────────────────────────────────┐
│      СЕРВЕР: content/pending/        │
└──────────────┬──────────────────────┘
               ↓ cron, раз в сутки
┌─────────────────────────────────────┐
│           ПУБЛИКАТОР                 │
│      autopilot/publish.js            │
│  pending → published, проставляет    │
│  даты, обновляет planned-urls.json   │
└──────────────┬──────────────────────┘
               ↓ (та же cron-команда)
┌─────────────────────────────────────┐
│           ЛИНКБИЛДЕР                 │
│      autopilot/linkbuilder.js        │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│               САЙТ                   │
│  Next.js читает content/published/   │
│  при первом запросе к URL, кеширует  │
│  результат на 24 часа (ISR)          │
└───────────────────────────────────────┘
```

Отдельно, независимо от контентного потока — деплой **кода**:
```
git push (site/, autopilot/-скрипты, docs/)
        ↓
GitHub Actions: git pull на сервере
        ↓
Если менялся site/ или autopilot/:
   npm install → npm run build → pm2 restart
```

## Два независимых канала синхронизации

Это ключевое архитектурное решение, разделяющее код и контент:

**Git — только код.** Отслеживает `site/`, `autopilot/` (сами скрипты,
не их вывод), `docs/`, `content/semantic_clusters.json`. `git pull` на
сервере безопасен всегда — в этих файлах сервер сам ничего не меняет,
конфликтов быть не может.

**Rsync — только контент.** Файлы, которые `publish.js`/`linkbuilder.js`/
`build-anchors.js` мутируют прямо на сервере (`content/pending/`,
`content/published/`, `content/anchors.json`,
`site/content/planned-urls.json`), исключены из git через `.gitignore`.
Доставка новых статей — `scripts/sync-content.sh`, обычный `rsync` по SSH,
не связанный с git вообще.

Причина разделения: если бы контент был в git, сервер после публикации
статьи имел бы незакоммиченные изменения в рабочей копии (перемещённые
файлы, переписанный `planned-urls.json`), и следующий `git pull`
конфликтовал бы с этими изменениями. Разделив каналы, `git pull`
остаётся тривиальной операцией всегда, а публикация контента полностью
изолирована от процесса деплоя кода.

## Генерация статьи (ручной процесс)

```
Кластер из semantic_clusters.json
        ↓
   Промпт 2 (в чате с Claude) → JSON карточка
        ↓
   Промпт 3 (в чате с Claude) → готовая статья с текстом
        ↓
   Сохранить вручную в content/pending/{page_id-с-дефисами}.json (локально)
        ↓
   scripts/sync-content.sh → на сервер, в content/pending/
```

Автоматической валидации между Промптом 2 и Промптом 3 нет — `validator.js`
не реализован. Проверка качества (чек-лист `skill_06_checklist.md`)
выполняется тем же диалогом с Claude на этапе генерации.

Отдельно запланирован `autopilot/validate-clusters.js` — проверка самого
`semantic_clusters.json` (не отдельных статей) перед коммитом, на предмет
опечаток в `url` (дефис vs подчёркивание), дублей `id`, отсутствия
обязательных полей. Обоснование — реальный прецедент: опечатка в `url`
одного кластера уже делала готовую статью «невидимой» для
`update-planned-urls.js` без явной ошибки где-либо в пайплайне.

## Публикация и линковка (реализованы, работают отдельно друг от друга)

**Публикация** — `autopilot/publish.js`, запускается по cron на сервере
раз в сутки. Берёт N старейших файлов из `content/pending/` (FIFO по
времени изменения файла), проставляет `date_published`/`date_modified`,
перекладывает в `content/published/`, пересобирает
`site/content/planned-urls.json`.

**Линковка** — `autopilot/linkbuilder.js`, запускается той же cron-командой
следом за публикацией. Проставляет контекстные внутренние ссылки по
словарю анкоров.

Оба скрипта проверены end-to-end вручную на реальном прогоне публикации
батчем, публикации по одной статье, линковки каждого файла и прод-сборки
сайта.

## Хранилище

Реализация — JSON-файлы, читаются и пишутся напрямую через `fs` в каждом
скрипте по отдельности (`publish.js`, `update-planned-urls.js`,
`linkbuilder.js`, `build-anchors.js`, роут `app/[...slug]/page.tsx`,
`app/sitemap.ts`). Общего интерфейса хранилища (`storage.js`) нет —
см. `STORAGE.md`.

Важно: любой файл, который должен читаться в рантайме с актуальным
содержимым (а не «замороженным» на момент сборки), нужно читать через
`fs.readFile`, а не через статический `import ... from "*.json"`.
Прецедент: `sitemap.ts` изначально был написан со статическим импортом
`planned-urls.json` и не показывал новые публикации вплоть до полной
пересборки; исправлено переходом на `fs.readFile`.

```
/content/
  semantic_clusters.json  ← В GIT
  /pending/      ← НЕ в git, доставляется rsync'ом
  /published/    ← НЕ в git, создаётся публикатором на сервере
  /error/        ← НЕ в git, не используется
  /images/       ← НЕ в git
```

## Сайт и кеширование (ISR)

Next.js читает статьи из `/content/published/`.
Роут `/[...slug]/page.tsx` использует on-demand ISR:
- `generateStaticParams()` возвращает пустой массив (конкретные URL не
  известны на этапе сборки)
- `revalidate = 86400` (сутки)

Первый запрос к любому URL рендерит страницу «вживую» (читает JSON,
рендерит React-дерево) и кеширует результат. Все последующие запросы к
этому же URL в течение суток отдаются из кеша мгновенно — без чтения
диска и без повторного рендера. Подтверждено заголовками ответа:
`x-nextjs-cache: MISS` на первый запрос, `HIT` — на все последующие,
`Cache-Control: s-maxage=86400, stale-while-revalidate`.

Без `generateStaticParams` `revalidate` на каталог-роут не действует —
Next.js трактует такой роут как полностью динамический (проверено
эмпирически до и после добавления этой функции).

`sitemap.xml` работает по тому же принципу: `revalidate = 3600` (час),
но обязательно поверх `fs.readFile`, не статического импорта (см. выше).
Новая публикация появляется в sitemap не сразу, а с задержкой до часа.

Следствие для деплоя: пересборка сайта (`npm run build`) нужна только
при изменении кода, не при публикации контента. CI/CD именно поэтому
разделён на два потока (см. выше).
