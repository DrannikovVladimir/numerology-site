# Контентный пайплайн

## Общая схема

```
Кластер из semantic_clusters.json
        ↓ (вручную, в чате с Claude)
   Промпт 2 → JSON карточка
        ↓ (вручную, в чате с Claude)
   Промпт 3 → готовая статья
        ↓
   Сохранить вручную в content/pending/{page_id-с-дефисами}.json
        ↓
   autopilot/publish.js → content/published/
        ↓
   autopilot/linkbuilder.js  ← отдельный ручной шаг, до или после публикации,
                                не вызывается автоматически ни одним скриптом
```

Автоматической валидации между Промптом 2 и Промптом 3 нет —
`autopilot/validator.js` не реализован. Проверка по чек-листу
(`skill_06_checklist.md`) выполняется тем же диалогом с Claude на этапе
генерации.

Перед коммитом изменений в `semantic_clusters.json` — прогонять
`node autopilot/validate-clusters.js` (проверяет дефис/подчёркивание в
`url`, дубли `id`, обязательные поля).

Если статья уже опубликована и её нужно доработать (добавить картинку,
раздел и т.п.) — это отдельный процесс, не через `content/pending/`.
См. `docs/ARCHITECTURE.md` и `scripts/update-article.sh`: файл правится
локально и доставляется на сервер перезаписью (не через очередь
публикации), с точечным сбросом кеша страницы.

## Промпт 2 — карточка

**Вход:** номер или название кластера (агент сам ищёт его в
`semantic_clusters.json` и берёт `page_type`, `url`, `primary_keyword`,
`template`, `hub_id`)
**Выход:** JSON карточка страницы

Карточка содержит:
- Мета-данные (title, description, h1)
- Структуру контента (blocks с описаниями)
- LSI-ключи из семантики кластера
- FAQ вопросы и ответы
- Внутренние ссылки
- word_count_target

### Правила заполнения `hub_id` и анкора крошки на хаб (важно, было источником багов)

В `semantic_clusters.json` поле `hub_id` хранится **через подчёркивание**
(например `chislo_sudby`). Промпт 2 обязан конвертировать все подчёркивания
в дефисы перед вставкой в карточку статьи: `chislo_sudby` → `chislo-sudby`.
Раньше формулировка правила была противоречивой («бери точно» + «но через
дефис»), из-за чего конвертация выполнялась непоследовательно — часть уже
опубликованных статей содержит `hub_id` с подчёркиванием. Правило
переформулировано явно, актуальные генерации должны быть чистыми.

Анкор ссылки вверх (`internal_links.up.anchor`, текст хлебной крошки на
хаб) теперь берётся **дословно** из поля `title` соответствующего
hub-кластера в `semantic_clusters.json`, а не придумывается заново в
каждой статье. Раньше анкор генерировался свободно, из-за чего регистр
текста крошки был непоследователен между сериями (например «числа
судьбы» со строчной у одной серии против «Ангельские числа» с заглавной
у другой). Правило теперь явное в Промпте 2, раздел «Перелинковка».

## Промпт 3 — генератор статьи

**Вход:** валидный JSON из Промпта 2
**Выход:** готовая статья + финальный JSON с реальным контентом

Статья содержит:
- Текст по всем блокам из карточки
- TOV: мистический экспертный
- Ключевые слова распределены равномерно
- Внутренние ссылки на своих местах (промпт 3 сам расставляет их вручную по
  правилам `skill_06_checklist.md` — `linkbuilder.js` дальше по пайплайну не
  заменяет эту работу, а лишь добавляет ссылки в те H2-разделы, где промпт 3
  ни одной не поставил)
- CTA placeholders
- FAQ с переформулированными вопросами

Поля `date_published`/`date_modified` промпт не создаёт — они появляются
только при публикации (`autopilot/publish.js`).

## Правила контента

**TOV — мистический экспертный:**
- Lead после H2: 3-4 предложения, 70-90 слов
- Абзацы после H3: два абзаца, между ними fact или quote (не чаще чем в каждом втором H3)
- Числа как живые сущности
- Обращение на "вы"

**Запрещено:**
- Арифметика с конкретными числами в тексте
- AI-клише (включает в себя, уникальный, древняя мудрость...)
- Дословное копирование поисковых запросов

**Обязательно:**
- CTA на Telegram-бот в разделах с расчётами
- Минимум: 1 fact, 1 quote, 1 table, 1 list на статью
- fact_row — ровно 1 раз в первом разделе H2
- 1-2 контекстные ссылки в каждом H2-разделе, приоритет уже опубликованным
  URL (`planned-urls.json`)

## Чек-лист качества

Чек-лист `skill_06_checklist.md` применяется вручную в процессе генерации
(тем же диалогом с Claude), не отдельным автоматическим прогоном.

## Приоритет генерации

```
1. Standalone (13 страниц)   ← первый батч
2. Spoke — числа судьбы      ← второй батч
3. Spoke — часы              ← третий батч
4. Spoke — ангельские числа  ← четвёртый батч
5. Spoke — месяцы            ← пятый батч
6. Spoke — совместимость     ← шестой батч
```
Ориентир по порядку, не жёсткое расписание — на практике генерация шла
вперемешку по сериям.

Hub страницы пишутся вручную до запуска остального контента.
